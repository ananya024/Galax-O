SECTION 1 — PROJECT OVERVIEW
====================================================================

1.1 What this project is
Galax-O is a modern, high-performance, real-time multi-user chat application designed to support sub-second messaging capabilities. The application features a decoupled, full-stack architecture comprising:
  * A stateful, real-time, bi-directional backend server powered by NestJS (v11) and TypeScript.
  * A modular, reactive frontend interface implemented in React (v19) and Vite.
  * A relational Postgres database instance accessed via the TypeORM Object-Relational Mapper (ORM) to handle data persistence, indexing, and relationships.
  * Stateful, bi-directional Socket.IO gateways to coordinate real-time events, user presence indicators, message routing, and delivery checkmarks.

1.2 Overall Objective
The primary objective of Galax-O is to create a secure, reliable, and responsive direct instant messaging solution. It provides a structured, enterprise-ready blueprint for a messaging system that implements security best practices (bcrypt hashing, JWT authentication guards, input validation pipes) and real-time synchronization patterns, while avoiding the latency, network overhead, and concurrency bugs typical of basic web communication architectures.

1.3 Real-World Problem It Solves
Traditional web collaboration tools suffer from communication lag due to the standard stateless HTTP request-response model, where client browsers must initiate all traffic. In business coordination, social networking, and support desk applications, this latency is unacceptable. Galax-O bridges this gap by establishing stable, low-overhead TCP streams that allow servers to push data to clients instantly as soon as events occur.

1.4 Why a Real-Time Chat Application is Needed
Communication velocity dictates operational efficiency. Applications like customer support, remote teamwork coordination, and transactional alert delivery require immediate message propagation. Standard REST endpoints cannot push alerts to an offline or waiting browser without resorting to resource-heavy polling. A stateful WebSocket connection is mandatory to achieve immediate text routing with minimal packet headers.

1.5 Functional Requirements
  * User Lifecycle: User registration with unique usernames, secure logins with credential verification, and profile retrieval.
  * Direct Messaging (1-to-1): Real-time text routing between active users, with automatic persistence to the Postgres database.
  * Presence Management: Real-time user list showing who is online/offline with instant status broadcasts upon connection state changes.
  * Message Receipts: Dynamic status indicators (single tick for sent, double ticks for delivered to socket, blue ticks for read by receiver).
  * System Auto-Replies: Automated greeting mechanisms ("Thank you for the message") sent once daily per active chat pair, alongside system alerts informing senders when their target is offline.

1.6 Non-Functional Requirements
  * Sub-Second Latency: Average message transmission time under 100ms.
  * Data Persistence: Guaranteed chronological message archival and storage.
  * Security: Hashed passwords, stateless API routing, and handshake verification.
  * Modularity & Scalability: Clean, decoupled code structure, separating HTTP APIs, persistent schemas, and socket gateways.
  * Responsive Layout: A fluid, glassmorphic UI matching mobile, tablet, and desktop screens.

1.7 Major Features
  * Bcrypt-reinforced signup/login.
  * JWT Stateless Authorization.
  * Event-driven Socket.IO direct messaging.
  * Live status broadcasts and user registry updates.
  * Active unread counters per sender.
  * Dynamic visual checkmarks (Sent, Delivered, Read).
  * Render boot-startup detection (ServerLoader) to mitigate cold starts.

1.8 End Users
  * General users looking for a clean, secure chat client.
  * Internship and college evaluators looking for a structured, production-grade project documentation base.
  * Future developers seeking a reference architecture for NestJS + TypeORM + Socket.IO + React.

1.9 High-Level Architecture
Galax-O utilizes a multi-tier, decoupled architecture:
  1. Client Layer (React/Vite): Handles UI rendering, browser session storage, state management, Axios API dispatching, and Socket.IO event listeners.
  2. Gateway/API Layer (NestJS Controllers & Gateways): Exposes REST endpoints for stateless actions and manages persistent WebSocket connections for stateful streams.
  3. Service Layer (NestJS Services): Implements the core business logic, validation rules, password hashing comparisons, and queries.
  4. Persistence Layer (TypeORM/PostgreSQL): Maps entities to SQL tables, runs transactional updates, and handles relational database storage.

1.10 Complete Technology Stack
  * Frontend: React 19, Vite, Axios, Socket.IO Client, React Router DOM 7, Vanilla CSS.
  * Backend: NestJS 11, TypeScript, Socket.IO, TypeORM, `@nestjs/jwt`, Passport JWT, bcrypt.
  * Database: PostgreSQL, `pg` driver.
  * Tools: Git, GitHub, Swagger-UI, Postman, Vercel (UI host), Render (API & DB host).

1.11 Overall Request Flow
  * Stateless Request: Client -> Axios Interceptor (attaches JWT) -> NestJS Controller -> AuthGuard -> Users/Messages Service -> TypeORM Repository -> PostgreSQL -> JSON Response back to Client.
  * Stateful Request: Client -> Handshake (token verification) -> WebSocket Connection -> emit("private-message") -> ChatGateway -> MessagesService (persists to SQL) -> Recipient WebSocket ID lookup -> emit("private-message") to recipient -> Client updates state -> UI re-renders with message bubble.

1.12 Why the Chosen Architecture Was Used
  * Modularity: NestJS enforces structural patterns (controllers, services, modules) preventing spaghetti code. Gaining dependency injection for free simplifies testing and decoupling.
  * Performance: Node.js's asynchronous, non-blocking single-threaded event loop handles thousands of concurrent WebSocket connections efficiently, unlike multi-threaded platforms which suffer from high thread overhead.
  * Data Integrity: PostgreSQL ensures ACID compliance, allowing reliable message ordering via auto-incrementing timestamps and relational constraints.

2.1 Architecture Diagram & Component Explanations
The flow from the browser client down to the database layers operates as follows:

```
[ Client Browser: React / Vite SPA ]
       |                  |
       | Axios HTTP       | WebSocket (WSS)
       | (REST APIs)      | (Stateful Events)
       v                  v
[ NestJS Controllers ] [ NestJS ChatGateway ]
       |                  |
       +--------+---------+
                |
                v
       [ NestJS Services ]
                |
                v
       [ TypeORM ORM Layer ]
                |
                v
       [ PostgreSQL Database ]
```

  * React: Implements a virtual DOM, updating the UI dynamically when new socket events occur without full page reloads.
  * NestJS: Acts as the unified controller-service router, applying validation pipes, CORS policies, and auth guards to secure incoming streams.
  * PostgreSQL: Relational storage engine that maintains schemas, runs indexes, and guarantees transaction logs.
  * TypeORM: Abstracts raw SQL generation, preventing SQL injections and transforming relational rows into typed TypeScript objects.
  * JWT: Decouples server memory from session tables. The backend verifies incoming signatures mathematically using public/secret keys without querying database sessions.
  * Socket.IO: Resolves browser compatibility issues, handling protocol upgrades and connection drops cleanly.

2.2 REST APIs and WebSockets Working Together
In Galax-O, REST APIs and WebSockets are complementary:
  * REST APIs handle slow, transactional operations where metadata overhead is acceptable (e.g., user signup, session creation, loading historical chat archives).
  * WebSockets handle fast, real-time message forwarding and presence updates where low latency is critical, utilizing the persistent TCP connection established during the handshake.

2.3 Complete Lifecycle of a User
  1. Signup: Client registers a username. NestJS hashes the password and saves it to PostgreSQL.
  2. Login: Client submits credentials. NestJS validates them and returns a signed JWT token.
  3. Handshake & Connect: The client saves the token in `sessionStorage` and connects to the Socket.IO gateway, passing the token in the handshake configuration.
  4. Online Status: The gateway verifies the token, marks the user as online, and broadcasts the status update to all connected clients.
  5. Fetching History: The client calls the REST API to load previous chat history.
  6. Message Flow: User A types a message and sends it via WebSockets. The gateway receives the message, saves it to the database, looks up User B's active socket, and routes the message directly.
  7. Receipts: User B receives the message, sending a delivery/read receipt back to the server, which updates the database and User A's screen dynamically.
  8. Disconnect: When User A closes the tab, the server detects the socket drop, updates the presence state, and broadcasts a "user-offline" event.

5.1 BACKEND FUNCTIONS

  * bootstrap() [main.ts]
    * Purpose: Initializes the NestJS server application.
    * Steps: Creates the app instance, enables CORS, registers global validation pipes and exception filters, generates Swagger documentation, and starts the server on the designated port.

  * canActivate(context: ExecutionContext) [auth.guard.ts]
    * Purpose: Validates incoming HTTP requests by verifying authorization tokens.
    * Steps: Extracts the request object, checks the Authorization header, validates the JWT token, and attaches the payload data to `request.user`. Returns `true` if valid, otherwise throws an UnauthorizedException.

  * signIn(userName: string, pass: string) [auth.service.ts]
    * Purpose: Authenticates users and returns signed JWT tokens.
    * Steps: Fetches the user by username, compares the password hash, and signs a JWT payload (containing user ID and username) if valid.

  * create(createUserDto: CreateUserDto) [users.service.ts]
    * Purpose: Registers new users in the database.
    * Steps: Hashes the plain text password using bcrypt, instantiates a User entity, and saves it. Throws a ConflictException if the username already exists.

  * authUser(username: string, password: string) [users.service.ts]
    * Purpose: Validates user credentials.
    * Steps: Queries the user table to retrieve the hashed password, and compares it with the plain text input. Returns the user object if verified, otherwise returns null.

  * handleConnection(client: Socket) [chat.gateway.ts]
    * Purpose: Handles new WebSocket connections.
    * Steps: Extracts the token from the handshake auth payload, verifies it, updates the online users map, and broadcasts presence updates to connected clients. If verification fails, it disconnects the socket.

  * handleDisconnect(client: Socket) [chat.gateway.ts]
    * Purpose: Handles socket disconnections.
    * Steps: Identifies the disconnected user, deletes their socket ID from the maps, and broadcasts updated online user lists to all clients.

  * handleMessage(client: Socket, payload: SendMessageDto) [chat.gateway.ts]
    * Purpose: Routes private messages between users.
    * Steps: Verifies the sender, persists the message to the database, checks the recipient's online status, and forwards the message to the active recipient socket. Sends system notifications to the sender if the recipient is offline.

  * create(senderId: string, createMessageDto: CreateMessageDto) [messages.service.ts]
    * Purpose: Persists messages in PostgreSQL.
    * Steps: Fetches sender and receiver entities, creates a new Message entity, and saves it. Returns a formatted data transfer object.

  * userhistory(senderId: string, receivername: string) [messages.service.ts]
    * Purpose: Retrieves conversation history between two users and updates read receipts.
    * Steps: Fetches the messages, sets `isRead` and `isDelivered` flags to true for incoming messages, updates the records in the database, and returns the message list.

  * unreadCount(loggedInUserId: string) [messages.service.ts]
    * Purpose: Aggregates unread message counts.
    * Steps: Retrieves all unread messages for the user, groups them by sender username, and returns the grouped count object.

5.2 FRONTEND FUNCTIONS

  * connectSocket(token) [socket.js]
    * Purpose: Configures and initializes the Socket.IO client instance.
    * Steps: Establishes a WebSocket connection with the backend URL, passes the auth token, and registers default lifecycle listeners.

  * App() [App.jsx]
    * Purpose: Main application component managing state and routing.
    * Steps: Checks if the backend is awake, rendering a loader if offline, and routes verified users to their respective pages.

  * Chat() [Chat.jsx]

  * handlePrivateMessage(msg) [Chat.jsx]

    * Steps: If the message is from the active conversation, it appends it to the chat history. Otherwise, it increments the unread count for that sender in the contact list.

6.1 React Architecture and State Hook Management
The React frontend is built as a single-page application (SPA) powered by Vite.
  * State Management: Managed locally using React hooks (`useState`, `useRef`, `useEffect`). Component interactions rely on refs (`selectedUserRef` and `meRef`) to manage mutable state within async socket listener callbacks, avoiding stale closures.
  * Routing: React Router DOM (v7) coordinates the page routes (`/login`, `/register`, `/chat`). Protected routes redirect unauthenticated users to `/login`, while public routes prevent logged-in users from accessing login screens.
  * API Layer: Axios interceptors automatically inject the JWT token into headers, decoupling request logic from components.
  * Socket Integration: The client socket is initialized upon successful authentication, registering event handlers to dynamically update states when other users connect or send messages.

6.2 UI Rendering Lifecycle and Stylesheets
  * Cold Start Loader: Displays a spinner until the backend responds to the `/users/loaded` endpoint, handling Render's free tier cold starts.
  * Responsive Layout: CSS grid layouts allow the UI to adapt dynamically. The layout splits into a sidebar list of users and a main chat viewport, which adjusts on mobile screens to show one panel at a time.

SECTION 7 — BACKEND ARCHITECTURE

7.1 NestJS Modular Architecture
NestJS provides a modular framework for structured backend development.
  * Modules: Define feature boundaries. `AppModule` acts as the root, importing feature-specific modules like `UsersModule`, `AuthModule`, `MessagesModule`, and `ChatModule`.
  * Controllers: Handle HTTP request routing, parse parameter decorators (`@Body`, `@Param`), and return JSON responses.
  * Services (Providers): Contain the business logic, database queries, and third-party integrations, injected into controllers and gateways via Dependency Injection (DI).
  * Data Validation: NestJS uses ValidationPipe combined with class-validator decorators in DTO classes to validate incoming payloads before executing code.
  * Filters and Guards: Exception filters catch errors globally to return standardized error responses. Auth guards verify JWT signatures on protected routes.

SECTION 8 — DATABASE DESIGN
====================================================================

8.1 Relational Schema and Entity Mapping
The Postgres database is mapped using TypeORM entities:
  * User Entity: Maps to the `user` table. It contains a unique `userId` (UUID primary key), a unique `username`, and a password column configured with `select: false` to hide hashes by default.
  * Message Entity: Maps to the `message` table. It contains a `messageId` (UUID primary key), a text content column, status flags (`isRead`, `isDelivered`), and a `createdAt` timestamp.
  * Relationships: Defined using `@ManyToOne` and `@OneToMany` decorators. A user has a 1-to-many relationship with messages, mapping to `sender` and `receiver` columns on the message table.

```
+------------------+                   +---------------------+
|      user        |                   |      message        |
+------------------+                   +---------------------+
| PK | userId      |1 <-------------\  | PK | messageId      |
|    | username    |                 \-|-FK | senderUserId   |
|    | password    |1 <-----------------FK | receiverUserId  |
+------------------+                   |    | content        |
                                       |    | isRead         |
                                       |    | isDelivered    |
                                       |    | createdAt      |
                                       +---------------------+
```

SECTION 9 — REST APIs
====================================================================

Galax-O's REST API endpoints are documented below:

9.1 User Registration (`POST /users`)
  * Purpose: Registers new accounts.
  * Auth Required: No.
  * Request Body: `CreateUserDto` `{ username, password }`.
  * Response Body: User object `{ userId, username }`.
  * Validation: Verifies username (3-30 chars) and password (3-100 chars).
  * Exceptions: Returns a 409 Conflict error if the username is taken.

9.2 Login Session (`POST /auth/login`)
  * Purpose: Authenticates credentials.
  * Auth Required: No.
  * Request Body: `CreateUserDto` `{ username, password }`.
  * Response Body: Token response `{ statusCode: 201, access_token, userId, username }`.
  * Exceptions: Returns a 401 Unauthorized error if credentials are invalid.

9.3 User Profile (`GET /auth/profile`)
  * Purpose: Retrieves authenticated user info.
  * Auth Required: Yes.
  * Request Body: None.
  * Response Body: User payload `{ sub: userId, username }`.

9.4 All Registered Users (`GET /users`)
  * Purpose: Returns all registered accounts.
  * Auth Required: No.
  * Response Body: Array of user objects.

9.5 Conversation History (`GET /messages/:receiver`)
  * Purpose: Loads historical messages between two users and updates read receipts.
  * Auth Required: Yes.
  * Parameters: `receiver` username.
  * Response Body: Array of message records `{ messageId, sender, receiver, content, createdAt, isRead, isDelivered }`.

9.6 Unread Count Summary (`GET /messages/unread-counts`)
  * Purpose: Fetches unread message counts grouped by sender.
  * Auth Required: Yes.
  * Response Body: Count mapping `{ [senderUsername]: unreadCountNumber }`.

SECTION 10 — SOCKET.IO IMPLEMENTATION
====================================================================

10.1 WebSockets Protocol
WebSockets provide full-duplex TCP streams that eliminate HTTP header overhead. Galax-O uses Socket.IO to manage connection lifecycles, reconnects, and event-based message distribution.

10.2 Event Breakdown

  * Connection Handshake: The client initiates a connection, passing the JWT token. The gateway extracts, verifies the token, and links the socket ID to the user ID in the registry maps.
  * user-online: Broadcast to all connected users when a client successfully connects.
  * online-users: Emits the updated list of online usernames to the client.
  * private-message: The client emits this event to send a message. The gateway persists the message, routes it to the recipient's socket if online, and returns delivery status updates.
  * message-delivered: Emitted to the sender's socket when a message is successfully delivered to the recipient.
  * user-offline: Broadcast to all clients when a user disconnects, removing their socket ID from the active maps.

SECTION 15 — DESIGN DECISIONS
====================================================================

15.1 Architectural Decisions and Trade-offs
  * NestJS vs Express: NestJS provides a structured, modular architecture with built-in dependency injection, which is cleaner and more maintainable than Express for larger systems.
  * Relational Database (PostgreSQL) vs Document Database (MongoDB): PostgreSQL is chosen for strict data integrity, table relationships, and transaction logs. MongoDB offers flexible schemas but lacks transactional guarantees.
  * TypeORM vs Raw SQL: TypeORM abstracts SQL generation, prevents SQL injections, and increases development speed. Raw SQL provides better query optimization control but increases code complexity.
  * WebSockets vs HTTP Polling: WebSockets enable bi-directional, real-time messaging with low overhead, whereas polling creates excessive request traffic and latency.
  * Session Storage vs HTTPOnly Cookies: sessionStorage is simple to implement but vulnerable to XSS. Production environments should use secure, HTTPOnly, SameSite cookies to protect tokens.
